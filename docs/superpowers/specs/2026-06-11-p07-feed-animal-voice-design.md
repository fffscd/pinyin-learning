# 喂动物语音输出玩法 设计

日期：2026-06-11
状态：方向已确认，细节待评审；第二阶段（识别）另行立项

## 背景

现有全部玩法都是「听音选卡」的输入型练习，认读关的产出环节也只是自评。缺少促使孩子开口的机制。

推测依据：语音产出练习对发音内化的作用大于纯辨听；「假装喂养」是 4–5 岁强偏好的游戏图式。通行结论，未逐条核验文献。

## 目标

新增花园玩法「喂动物」：屏幕上小动物面前摆一张拼音卡，孩子对着设备读出来，动物「听到了」就吃掉食物并表示满足。第一阶段只检测发声，不做识别——目标是建立「看到拼音就开口读」的习惯。

## 约束

- 麦克风权限是家长决策：玩法默认隐藏，须在家长区（p15）开启；浏览器权限弹窗由家长操作。
- 权限被拒或设备无麦克风时玩法入口不出现，零报错。
- 录音数据不上传、不存储：只取实时音量包络，用完即弃。HTTPS 或 localhost 才有 `getUserMedia`，GitHub Pages 满足。

## 设计

### 发声检测（第一阶段）

- `navigator.mediaDevices.getUserMedia({ audio: true })` → `AudioContext` + `AnalyserNode`，每帧取 RMS 音量。
- 判定：先采样 1 秒环境底噪取基线；之后 RMS 超过基线一定倍数且持续 ≥ 400ms 视为「开口读了」。阈值参数集中为常量便于调试。
- 防呆：拍桌子、喊叫同样会通过——第一阶段接受这一点，机制目标是开口而非正确，正确性由认读关与家长抽查（p05）保障。

### 玩法流程

1. 出现动物（兔子/熊猫等，SVG）+ 食物 + 拼音大卡，播一遍标准读音作示范。
2. 提示动画：动物侧耳倾听。孩子读出声 → 食物飞进动物嘴里，动物开心动效 + 既有 `correct` 音；记一颗星。
3. 5 秒内未检测到发声 → 再播一遍示范音，动物做「张大耳朵」动效继续等；不判错、不计错。
4. 每轮 5 题，题池用 `getUnlockedReadIds`（与认读关一致）。
5. 答题结果以 `source: "voice-attempt"` 记入练习次数，不写 Leitner 盒子（发声不等于读对）。

### 入口

- `GARDEN_GAME_MODES` 增加 `mode: "feed"`（图标：动物），仅当 `state.progress.settings.micEnabled === true` 且运行时权限可用才渲染。
- 家长区开关写入 `progress.settings.micEnabled`，`normalizeProgress` 默认 false。

### 第二阶段（仅记录方向，不在本期）

- 评估端侧识别：浏览器 `SpeechRecognition` 对幼儿普通话拼音单音节的可用性需实测，预期可靠性不足（推测，依据是该 API 面向连续语音且无儿童声学模型）；备选是音高/共振峰特征做四声粗判。实测结论出来前不动工。

## 不做的事（YAGNI）

- 不做语音识别、不做发音评分。
- 不录制、不上传、不缓存任何音频数据。
- 不在儿童端出现权限相关文案。

## 影响的文件

- `app.js`：`GARDEN_GAME_MODES`、新增 `buildFeedPracticeQuestions`、发声检测模块（独立函数组）、`questionStage` feed 分支、`startRound`、家长区开关。
- `styles.css`：动物、食物、倾听/进食动效。
- `assets/`：动物 SVG（可内联进 `app.js` 的 `illustration`/`icon` 体系）。
- `README.md`、`AGENTS.md` 测试清单。

## 验证方式

- headless：micEnabled=false 时花园不渲染 feed 入口；题目结构正确。
- 人工：真机（iPad/手机）权限授予、拒绝、无麦克风三条路径；底噪嘈杂环境（电视声）下的误触发率；孩子实测「读出声就喂到」的反馈是否及时（延迟应低于 500ms）。
